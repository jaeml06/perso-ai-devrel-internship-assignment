**Created**: 2026-03-10
**Status**: Draft

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 화이트리스트에 등록된 사용자의 Google 로그인 (Priority: P1)

허용된 이메일(예: kts123@estsoft.com)을 가진 사용자가 Google OAuth로 로그인하면 서비스의 더빙 기능에 정상 접근할 수 있다.

**Why this priority**: 인가된 사용자만 서비스에 접근하는 것이 핵심 보안 요구사항이며, 서비스 이용의 전제 조건이다.

**Independent Test**: Google 계정으로 로그인 후 대시보드(더빙 생성 페이지)에 접근 가능한지 확인하여 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 화이트리스트에 등록된 이메일의 Google 계정 사용자, **When** 로그인 페이지에서 Google 로그인 버튼 클릭 후 인증 완료, **Then** 메인 대시보드 페이지로 리다이렉트되어 서비스를 이용할 수 있다.
2. **Given** 이미 로그인된 인가 사용자, **When** 보호된 페이지에 재방문, **Then** 세션이 유지되어 별도 로그인 없이 접근 가능하다.

---

### User Story 2 - 화이트리스트에 없는 사용자의 접근 차단 (Priority: P1)

화이트리스트에 등록되지 않은 이메일의 Google 계정으로 로그인을 시도하면, 서비스 이용이 차단되고 안내 메시지가 표시된다.

**Why this priority**: 비인가 사용자 차단은 보안의 핵심이며, P1 로그인 기능과 동등하게 중요하다.

**Independent Test**: 화이트리스트에 없는 Google 계정으로 로그인 시도 후 차단 페이지가 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 화이트리스트에 없는 이메일의 Google 계정 사용자, **When** Google 로그인 완료, **Then** 접근 차단 페이지로 이동하며 "접근 권한이 없습니다" 안내 메시지가 표시된다.
2. **Given** 차단된 사용자, **When** 보호된 페이지 URL을 직접 입력하여 접근 시도, **Then** 로그인 페이지 또는 차단 페이지로 리다이렉트된다.

---

### User Story 3 - 로그인 페이지 진입 (Priority: P2)

비로그인 사용자가 서비스에 처음 방문하면 로그인 페이지가 표시되고, Google 로그인 버튼을 통해 인증을 시작할 수 있다.

**Why this priority**: 로그인 UI는 인증 플로우의 진입점으로 필수적이나, 백엔드 로직보다 후순위.

**Independent Test**: 비로그인 상태에서 서비스 접근 시 로그인 페이지가 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 비로그인 사용자, **When** 서비스 루트 URL 또는 보호된 페이지에 접근, **Then** 로그인 페이지가 표시되고 Google 로그인 버튼이 보인다.
2. **Given** 비로그인 사용자, **When** 로그인 페이지에서 Google 로그인 버튼 클릭, **Then** Google OAuth 인증 화면으로 이동한다.

---

### User Story 4 - 로그아웃 (Priority: P3)

로그인된 사용자가 로그아웃 버튼을 클릭하면 세션이 종료되고 로그인 페이지로 이동한다.

**Why this priority**: 기본적인 세션 관리 기능이나, 핵심 인증 플로우보다 후순위.

**Independent Test**: 로그인 상태에서 로그아웃 버튼 클릭 후 세션 종료 확인.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** 로그아웃 버튼 클릭, **Then** 세션이 종료되고 로그인 페이지로 리다이렉트된다.

---

### Edge Cases

- 화이트리스트 DB에 연결할 수 없는 경우 → 안전하게 접근을 차단하고 에러 안내 표시
- Google OAuth 인증 과정에서 사용자가 취소한 경우 → 로그인 페이지로 돌아감
- 세션이 만료된 상태에서 보호된 페이지에 접근하는 경우 → 로그인 페이지로 리다이렉트
- 동일 이메일로 중복 로그인 시도 → 기존 세션이 유지되거나 갱신됨

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 Google OAuth 2.0을 통한 사용자 로그인을 지원해야 한다.
- **FR-002**: 시스템은 로그인한 사용자의 이메일을 허용 리스트(화이트리스트) DB와 대조하여 접근 권한을 판단해야 한다.
- **FR-003**: 화이트리스트에 등록된 사용자만 서비스의 보호된 페이지(더빙 대시보드 등)에 접근할 수 있어야 한다.
- **FR-004**: 화이트리스트에 없는 사용자에게는 접근 차단 안내 페이지를 표시해야 한다.
- **FR-005**: 화이트리스트 데이터는 Turso 데이터베이스에 저장 및 관리해야 한다.
- **FR-006**: `kts123@estsoft.com` 이메일은 화이트리스트에 반드시 포함되어야 한다.
- **FR-007**: 비로그인 사용자가 보호된 페이지에 접근 시 로그인 페이지로 리다이렉트해야 한다.
- **FR-008**: 사용자는 로그아웃하여 세션을 종료할 수 있어야 한다.

### Key Entities

- **User (허용 사용자)**: 서비스 접근이 허용된 사용자. 주요 속성: 이메일, 이름, 등록일. 화이트리스트의 기본 단위.
- **Session (세션)**: 로그인한 사용자의 인증 상태. 주요 속성: 사용자 정보, 만료 시점. 인증 여부 판단에 사용.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 화이트리스트에 등록된 사용자가 10초 이내에 Google 로그인을 완료하고 서비스에 접근할 수 있다.
- **SC-002**: 화이트리스트에 없는 사용자가 로그인 시도 시 100% 접근이 차단되고 안내 메시지가 표시된다.
- **SC-003**: `kts123@estsoft.com` 계정으로 로그인 시 항상 정상적으로 서비스에 접근할 수 있다.
- **SC-004**: 비로그인 사용자가 보호된 페이지에 접근 시 100% 로그인 페이지로 리다이렉트된다.

## Assumptions

- 로그인 페이지는 미니멀 스타일로, 중앙 정렬된 카드에 서비스 로고 + Google 로그인 버튼만 표시한다.
- 인증 보호는 미들웨어 수준에서 적용하여, 기존 더빙 대시보드 코드 변경 없이 보호한다.
- 화이트리스트 관리는 DB 직접 조작(시드 데이터)으로 수행하며, 별도 관리자 UI는 제공하지 않는다.
