# Feature Specification: 로그인 오류 원인 구분 안내

**Feature Branch**: `feat/#15-auth-error-feedback`
**Created**: 2026-03-17
**Status**: Draft
**Input**: DB 오류로 로그인이 차단된 경우와 미등록 사용자가 차단된 경우가 동일한 오류 화면을 보여줘 사용자가 원인을 파악할 수 없는 문제

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 미등록 사용자에게 명확한 접근 거부 안내 (Priority: P1)

화이트리스트에 등록되지 않은 이메일로 로그인을 시도한 사용자가 "이 서비스는 허가된 사용자만 이용할 수 있습니다"라는 명확한 메시지를 받는다.

**Why this priority**: 정상적인 접근 거부 상황은 서비스의 기본 보안 요건으로, 사용자에게 정확한 이유를 전달해야 한다.

**Independent Test**: 미등록 이메일로 로그인 시도 → 접근 거부 화면에 "화이트리스트에 등록되지 않은 계정" 취지의 메시지가 표시되면 성공.

**Acceptance Scenarios**:

1. **Given** 화이트리스트에 없는 이메일로 Google 로그인을 시도했을 때, **When** 인증 시스템이 사용자 확인을 완료하면, **Then** "이 서비스는 허가된 사용자만 이용할 수 있습니다. 이메일 화이트리스트에 등록된 계정으로 로그인해 주세요." 메시지가 표시된다.
2. **Given** 미등록 사용자가 접근 거부 화면을 보고 있을 때, **When** "로그인 페이지로 돌아가기" 버튼을 클릭하면, **Then** 로그인 페이지로 이동한다.

---

### User Story 2 - 서버 오류 시 사용자 친화적 오류 안내 (Priority: P1)

화이트리스트에 등록된 사용자가 서버 내부 오류(예: DB 연결 실패)로 인해 로그인이 차단됐을 때, 미등록 사용자와 **다른** 메시지를 받아 서비스 문제임을 인지할 수 있다.

**Why this priority**: 현재 kts123@estsoft.com처럼 등록된 사용자가 서버 오류로 차단되어도 미등록 사용자와 동일한 메시지를 받아 혼란이 발생하고 있다. 원인 구분이 핵심 버그 수정 사항이다.

**Independent Test**: DB 연결이 실패하는 환경에서 등록된 이메일로 로그인 시도 → "서비스에 일시적 문제가 발생했습니다. 잠시 후 다시 시도해주세요." 취지의 메시지가 표시되면 성공.

**Acceptance Scenarios**:

1. **Given** 화이트리스트에 등록된 이메일로 로그인을 시도했고 서버 내부 오류가 발생했을 때, **When** 인증 시스템이 오류를 감지하면, **Then** "서비스에 일시적 문제가 발생했습니다. 잠시 후 다시 시도해주세요." 메시지가 표시된다.
2. **Given** 서버 오류 안내 화면을 보고 있는 사용자가, **When** 재시도 버튼을 클릭하면, **Then** 로그인 페이지로 이동하여 다시 시도할 수 있다.

---

### User Story 3 - 이메일 미제공 시 안내 (Priority: P2)

Google 계정에서 이메일 정보를 제공하지 않은 경우 사용자에게 원인을 안내한다.

**Why this priority**: 드문 케이스지만 처리하지 않으면 서버 오류와 동일하게 묶여 혼란을 야기한다.

**Independent Test**: 이메일 없이 OAuth 인증 완료 시 "이메일 정보를 확인할 수 없습니다" 취지의 메시지가 표시되면 성공.

**Acceptance Scenarios**:

1. **Given** Google 계정에서 이메일 정보가 제공되지 않은 상태로 로그인을 시도했을 때, **When** 인증 시스템이 이메일 부재를 감지하면, **Then** "Google 계정에서 이메일 정보를 가져올 수 없습니다. Google 계정 설정을 확인해주세요." 메시지가 표시된다.

---

### Edge Cases

- DB 연결이 간헐적으로 실패하는 경우 → 매 요청마다 서버 오류 메시지가 일관되게 표시되어야 함
- 로그인 직후 DB가 복구된 경우 → 재시도 시 정상 로그인 가능해야 함
- 동일한 사용자가 오류 발생 후 즉시 재시도 → 재시도 횟수 제한 없이 허용

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 미등록 사용자 차단과 서버 내부 오류로 인한 차단을 **서로 다른 메시지**로 사용자에게 안내해야 한다.
- **FR-002**: 서버 내부 오류로 로그인이 차단된 경우 "서비스에 일시적 문제가 발생했습니다. 잠시 후 다시 시도해주세요." 취지의 메시지를 표시해야 한다.
- **FR-003**: 미등록 사용자가 차단된 경우 "이 서비스는 허가된 사용자만 이용할 수 있습니다." 취지의 메시지를 표시해야 한다.
- **FR-004**: Google 계정에서 이메일 정보를 제공하지 않은 경우 별도 안내 메시지를 표시해야 한다.
- **FR-005**: 모든 오류 화면에는 사용자가 다시 시도할 수 있는 로그인 페이지 이동 버튼이 있어야 한다.
- **FR-006**: 서버 내부 오류(DB 연결 실패 등) 발생 시 **오류 유형과 발생 시각**을 서버 로그에 기록하여 운영자가 Vercel 로그에서 확인할 수 있어야 한다. 이메일 등 개인정보는 로그에 포함하지 않는다.

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 서버 오류로 차단된 사용자와 미등록으로 차단된 사용자가 **서로 다른** 오류 메시지를 받는다
- **SC-002**: 서버 오류 발생 시 사용자가 "잠시 후 재시도"라는 행동 지침을 메시지에서 확인할 수 있다
- **SC-003**: 미등록 사용자가 오류 메시지만으로 자신이 화이트리스트에 없어서 차단됐음을 이해할 수 있다
- **SC-004**: 등록된 사용자가 서버 오류 후 재시도하면 정상 로그인에 성공한다 (서버 복구 전제)
- **SC-005**: 서버 내부 오류 발생 시 오류 유형과 발생 시각이 Vercel 로그에서 확인 가능하다 (이메일 등 개인정보 미포함)

## Assumptions

- 서버 내부 오류 로깅은 이번 범위에 포함하며, 오류 유형과 타임스탬프만 기록한다 (이메일 등 개인정보 제외).
- 오류 유형은 `/unauthorized` 동일 URL에서 쿼리 파라미터(`?error=...`)로 구분하며, 파라미터 값에 따라 표시 메시지를 달리한다.
- 재시도 횟수 제한이나 계정 잠금 기능은 이번 범위에 포함하지 않는다.

## Clarifications

### Session 2026-03-17

- Q: 서버 오류와 미등록 사용자를 어떻게 화면에서 구분할까요? → A: 같은 URL(`/unauthorized`), 쿼리 파라미터(`?error=...`)로 구분
- Q: Vercel 로그에 기록할 정보 범위를 선택해주세요 → A: 오류 유형 + 타임스탬프만 기록 (이메일 미포함)
