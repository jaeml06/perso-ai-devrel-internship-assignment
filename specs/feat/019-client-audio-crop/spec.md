# Feature Specification: 클라이언트 측 1분 미디어 크롭

**Feature Branch**: `019-client-audio-crop`
**Created**: 2026-03-20
**Status**: Draft
**Input**: User description: "대용량 파일 업로드 시나리오에 대비해 1분 crop해서 처리하는 기능. 1분 크롭은 서버가 아닌 클라이언트 측에서 수행."

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 1분 초과 미디어 자동 크롭 (Priority: P1)

사용자가 1분을 초과하는 오디오 또는 비디오 파일을 업로드하면, 시스템이 자동으로 처음 1분만 잘라서 더빙 파이프라인에 전달한다. 크롭은 서버가 아닌 클라이언트(브라우저)에서 수행되므로 서버 부하 없이 대용량 파일을 처리할 수 있다.

**Why this priority**: 대용량 파일 업로드 시 서버 전송량과 처리 시간을 줄이는 핵심 기능이며, 사용자 경험의 근본적인 개선점이다.

**Independent Test**: 2분짜리 오디오 파일을 업로드하고, STT API로 전송되는 파일이 1분 분량인지 확인하면 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 사용자가 3분짜리 MP3 파일을 선택함, **When** 업로드 버튼을 누름, **Then** 클라이언트에서 처음 1분만 크롭된 파일이 더빙 파이프라인에 전달된다.
2. **Given** 사용자가 2분짜리 MP4 비디오를 선택함, **When** 업로드 버튼을 누름, **Then** 비디오의 처음 1분만 크롭되어 처리된다.
3. **Given** 사용자가 45초짜리 오디오 파일을 선택함, **When** 업로드 버튼을 누름, **Then** 크롭 없이 원본 파일 그대로 처리된다 (1분 이하이므로).

---

### User Story 2 - 크롭 진행 상태 피드백 (Priority: P2)

사용자가 대용량 파일을 업로드했을 때, 클라이언트 측 크롭이 진행 중임을 알 수 있는 시각적 피드백을 제공한다. 사용자는 현재 무엇이 진행되고 있는지 혼란 없이 인지할 수 있다.

**Why this priority**: 크롭 처리에 수 초가 소요될 수 있으며, 진행 상태 없이는 사용자가 앱이 멈춘 것으로 오인할 수 있다.

**Independent Test**: 대용량 파일 업로드 시 크롭 단계에서 진행 상태 메시지가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 1분 초과 파일을 업로드함, **When** 크롭이 진행 중일 때, **Then** "미디어를 1분으로 자르는 중..." 등의 상태 메시지가 표시된다.
2. **Given** 크롭이 완료됨, **When** 다음 단계(전사)로 진행될 때, **Then** 상태가 자연스럽게 전사 단계로 전환된다.

---

### User Story 3 - 크롭 알림 및 원본 길이 안내 (Priority: P3)

사용자가 1분을 초과하는 파일을 업로드했을 때, 원본 길이와 1분으로 잘릴 것이라는 안내 메시지를 사전에 보여준다. 사용자가 의도치 않게 파일이 잘리는 것을 방지한다.

**Why this priority**: 사용자 투명성 확보를 위한 부가 기능으로, 핵심 크롭 기능과 피드백이 먼저 구현되어야 한다.

**Independent Test**: 5분짜리 파일 선택 시 "원본 5분 중 처음 1분만 처리됩니다" 안내가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 5분짜리 파일을 선택함, **When** 파일이 선택된 직후, **Then** "원본 길이: 5분 00초 → 처음 1분만 처리됩니다" 안내 메시지가 표시된다.
2. **Given** 사용자가 30초짜리 파일을 선택함, **When** 파일이 선택됨, **Then** 크롭 관련 안내 메시지는 표시되지 않는다.

---

### Edge Cases

- 파일이 정확히 1분(60초)인 경우: 크롭 없이 원본 그대로 처리한다.
- 1분보다 약간 긴 경우 (예: 61초): 정상적으로 1분 크롭 적용한다.
- 크롭 처리 중 브라우저 탭을 닫거나 이동하는 경우: 처리가 중단되며 별도 복구 없이 재시도해야 한다.
- 손상된 미디어 파일의 경우: 크롭 실패 시 사용자에게 오류 메시지를 표시하고 원본 파일 재선택을 안내한다.
- 지원하지 않는 코덱의 미디어 파일: 크롭 실패 시 적절한 오류 메시지를 표시한다.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 업로드된 미디어 파일의 재생 시간을 클라이언트 측에서 감지해야 한다 (MUST).
- **FR-002**: 재생 시간이 1분(60초)을 초과하는 파일은 클라이언트에서 처음 1분으로 크롭해야 한다 (MUST).
- **FR-003**: 크롭된 파일은 원본과 동일한 형식(오디오→오디오, 비디오→비디오)을 유지해야 한다 (MUST).
- **FR-004**: 크롭 처리는 서버가 아닌 클라이언트(브라우저) 내에서 수행되어야 한다 (MUST).
- **FR-005**: 1분 이하 파일은 크롭 없이 원본 그대로 파이프라인에 전달되어야 한다 (MUST).
- **FR-006**: 크롭 진행 중 사용자에게 시각적 피드백을 제공해야 한다 (MUST).
- **FR-007**: 1분 초과 파일 선택 시 크롭 예정임을 사용자에게 사전 안내해야 한다 (SHOULD).
- **FR-008**: 크롭 실패 시 사용자에게 명확한 오류 메시지를 표시해야 한다 (MUST).

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 1분 초과 미디어 파일이 업로드되면, 더빙 파이프라인에 전달되는 미디어가 60초 이내여야 한다.
- **SC-002**: 크롭 처리가 클라이언트에서 완료되며, 서버로 1분 초과 원본 데이터가 전송되지 않아야 한다.
- **SC-003**: 1분 이하 파일의 경우, 기존 플로우와 동일하게 크롭 없이 처리되어야 한다.
- **SC-004**: 크롭 진행 중 사용자가 현재 상태를 인지할 수 있는 피드백이 제공되어야 한다.
- **SC-005**: 크롭 실패 시 사용자가 다음 행동을 취할 수 있는 안내가 제공되어야 한다.

## Clarifications

### Session 2026-03-20

- Q: 1분 초과 파일 업로드 시, 크롭 전에 사용자 확인을 받아야 하나요? → A: 자동 크롭. 안내 메시지만 표시하고 확인 없이 자동으로 크롭 후 처리한다.

## Assumptions

- 기존 프로젝트에서 이미 FFmpeg WASM을 클라이언트 측에서 사용하고 있으므로, 동일한 인프라를 활용하여 크롭을 수행한다.
- 크롭 기준은 항상 "처음부터 1분"이며, 사용자가 구간을 선택하는 기능은 범위에 포함하지 않는다.
- 기존 25MB 파일 크기 제한은 유지하되, 크롭을 통해 실제 처리량을 줄인다.
- 크롭된 미디어의 품질은 원본과 동일하게 유지한다 (재인코딩 시 동일 비트레이트).
