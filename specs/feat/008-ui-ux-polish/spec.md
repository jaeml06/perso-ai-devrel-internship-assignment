# Feature Specification: UI/UX 폴리싱 — 반응형 레이아웃 & 일관된 디자인 시스템

**Feature Branch**: `feat/#8-ui-ux-polish`
**Created**: 2026-03-10
**Status**: Draft
**GitHub Issue**: #8

## 배경 및 목적

현재 AI 더빙 서비스의 대시보드 페이지(DubbingDashboardPage 및 하위 컴포넌트)는 Tailwind CSS 스타일이 전혀 적용되어 있지 않아 사용자에게 깨진 UI로 보인다. 로그인 페이지와 미인가 페이지는 일부 스타일이 적용되어 있으나 전체 서비스에 일관된 디자인 언어가 부재하다. 이 기능은 모든 페이지에 일관된 디자인 시스템을 적용하고, 반응형 레이아웃·모션/애니메이션·로딩 및 에러 화면을 최적화하여 완성도 있는 UI를 구현한다.

---

## User Scenarios & Testing

### User Story 1 — 대시보드 페이지 스타일 완성 (Priority: P1)

사용자가 로그인 후 `/dashboard`에 진입하면, 파일 업로드 폼·언어 선택·음성 선택·제출 버튼이 보기 좋고 사용하기 편한 카드 레이아웃으로 표시된다.

**Why this priority**: 현재 대시보드는 스타일이 전혀 없어 서비스의 핵심 기능이 사용 불가 수준으로 보임. 가장 높은 시각적 영향력.

**Independent Test**: `/dashboard`를 방문하여 폼 UI가 카드 레이아웃으로 렌더링되고, 모바일(375px) 및 데스크톱(1280px)에서 레이아웃이 올바르게 보이는지 확인.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** `/dashboard` 진입, **Then** 중앙 정렬된 카드 레이아웃 내 파일 업로드 UI, 언어 선택 드롭다운, 음성 선택 컴포넌트, 더빙 생성 버튼이 스타일링되어 표시됨
2. **Given** 모바일 화면(375px), **When** `/dashboard` 진입, **Then** 모든 폼 요소가 단일 컬럼으로 스크롤 없이 접근 가능하게 표시됨
3. **Given** 데스크톱 화면(1024px 이상), **When** `/dashboard` 진입, **Then** 콘텐츠가 최대 너비 내 중앙 정렬되어 여백이 적절히 배분됨

---

### User Story 2 — 파이프라인 로딩 및 에러 화면 최적화 (Priority: P2)

더빙 생성 중 각 처리 단계(STT → 번역 → TTS)가 시각적으로 명확하게 표시되고, 에러 발생 시 사용자가 원인을 파악하고 재시도할 수 있는 최적화된 화면이 제공된다.

**Why this priority**: 처리 중 사용자 피드백이 없으면 서비스가 멈춘 것처럼 보임. 에러 시 재시도 UX도 필수.

**Independent Test**: 더빙 생성 버튼 클릭 후 파이프라인 진행 화면이 단계별로 시각적 진행상태를 표시하는지 확인. 에러 상태에서 에러 메시지와 재시도 버튼이 명확하게 표시되는지 확인.

**Acceptance Scenarios**:

1. **Given** 더빙 생성 진행 중, **When** `transcribing/translating/synthesizing` 상태, **Then** 현재 진행 단계가 강조 표시되고 스피너 또는 애니메이션이 활성화됨
2. **Given** 단계 완료, **When** 다음 단계로 전환, **Then** 완료된 단계는 체크 아이콘과 함께 완료 색상으로 표시됨
3. **Given** 에러 발생, **When** 에러 상태 표시, **Then** 명확한 에러 메시지와 스타일된 재시도 버튼이 표시됨
4. **Given** 더빙 완료, **When** `complete` 상태, **Then** 오디오 플레이어가 스타일링되어 나타남

---

### User Story 3 — 모든 페이지 디자인 일관성 (Priority: P3)

로그인 페이지, 미인가 페이지, 대시보드 페이지, 네비게이션이 동일한 색상 팔레트·타이포그래피·간격 시스템을 사용하여 통일된 서비스 경험을 제공한다.

**Why this priority**: 핵심 기능 구현 후 전체적인 브랜드 일관성 확보가 필요.

**Independent Test**: 로그인 → 대시보드 → 미인가 페이지를 순서대로 방문하여 색상·폰트·버튼 스타일이 일관되게 보이는지 시각적으로 확인.

**Acceptance Scenarios**:

1. **Given** 모든 페이지, **When** 페이지 간 이동, **Then** 브랜드 색상(primary/secondary)·폰트 크기 체계·버튼 스타일이 일관됨
2. **Given** 내비게이션 바, **When** 모든 페이지 표시, **Then** 브랜드명과 로그아웃 버튼이 동일한 스타일로 고정 표시됨
3. **Given** 에러/정보 메시지, **When** 어느 페이지에서든 표시, **Then** 동일한 색상 의미론(빨강=에러, 초록=성공, 파랑=정보)을 따름

---

### User Story 4 — 모션 및 전환 애니메이션 (Priority: P4)

버튼 호버·페이지 요소 등장·파이프라인 상태 전환에 부드러운 마이크로 인터랙션이 추가되어 서비스가 생동감 있게 느껴진다.

**Why this priority**: 기능 완성 후 경험 품질을 높이는 마지막 레이어.

**Independent Test**: 버튼에 마우스를 올렸을 때 호버 효과가 있는지, 파이프라인 단계 전환 시 상태 변화가 부드럽게 보이는지 확인.

**Acceptance Scenarios**:

1. **Given** 버튼, **When** 마우스 호버, **Then** 색상/스케일 전환 애니메이션(duration 150-200ms)이 표시됨
2. **Given** 파이프라인 상태 전환, **When** 단계 변경, **Then** 단계 표시기가 부드럽게 전환됨
3. **Given** 페이지 최초 로드, **When** 주요 컨텐츠 표시, **Then** 페이드인 또는 슬라이드업 등장 효과가 적용됨

---

### Edge Cases

- 오디오 파일 로딩 중 플레이어가 비활성 상태로 표시되어야 함
- 매우 긴 파일명은 잘림(truncation) 처리되어야 함
- 네트워크 오류로 음성 목록 로드 실패 시 재시도 버튼이 표시되어야 함
- 폼 유효성 검사 에러 메시지가 해당 필드 아래 명확하게 표시되어야 함
- `prefers-reduced-motion` 미디어 쿼리를 존중하여 모션을 비활성화할 수 있어야 함

## Requirements

### Functional Requirements

- **FR-001**: 대시보드 페이지의 DubbingForm, PipelineProgress, AudioPlayer 컴포넌트에 Tailwind CSS 스타일이 적용되어야 함
- **FR-002**: 모든 페이지가 모바일(375px) 및 데스크톱(1024px 이상)에서 레이아웃이 깨지지 않아야 함
- **FR-003**: 파이프라인 진행 중 현재 단계가 시각적으로 구별되어야 함 (스피너, 강조 색상 등)
- **FR-004**: 에러 상태에서 사용자가 읽을 수 있는 에러 메시지와 재시도 버튼이 표시되어야 함
- **FR-005**: 로그인 페이지, 미인가 페이지, 대시보드 페이지가 동일한 색상 팔레트와 타이포그래피를 사용해야 함
- **FR-006**: 버튼 인터랙션(호버, 포커스, 비활성화)에 시각적 피드백이 있어야 함
- **FR-007**: 파이프라인 단계 전환에 CSS 트랜지션이 적용되어야 함
- **FR-008**: 접근성 기준을 충족해야 함 (색상 대비 4.5:1 이상, focus 링 표시, aria-label)

### Assumptions

- 프로젝트에 이미 Tailwind CSS가 설정되어 있음 (globals.css 및 tailwind.config.ts 존재)
- 기존 CSS 변수(--background, --foreground, --border 등)를 디자인 토큰으로 활용
- 추가 UI 라이브러리(shadcn/ui, Radix 등)는 사용하지 않고 순수 Tailwind로 구현

## Success Criteria

### Measurable Outcomes

- **SC-001**: 대시보드 페이지에서 모든 인터랙티브 요소(폼, 버튼, 선택 컴포넌트)가 시각적으로 구별 가능하고 클릭 가능하게 보임
- **SC-002**: 모바일(375px)과 데스크톱(1280px) 양쪽에서 레이아웃 깨짐 없이 핵심 기능에 접근 가능
- **SC-003**: 로그인 → 대시보드 → 미인가 페이지 간 이동 시 색상·버튼·간격이 일관되게 보임
- **SC-004**: 파이프라인 진행 중 현재 단계를 즉시 인식할 수 있음 (3초 이내 상태 파악)
- **SC-005**: 에러 발생 시 사용자가 원인 메시지를 읽고 재시도 버튼을 1회 클릭으로 찾을 수 있음

## Clarifications

### Session 2026-03-10

- Q: 디자인 일관성 기준으로 기존 LoginPage/UnauthorizedPage 스타일을 기준점으로 삼을까요, 아니면 새 디자인 언어를 정의할까요? → A: 기존 LoginPage/UnauthorizedPage 스타일(흰 카드, 그레이 배경, 블루 primary)을 기준으로 대시보드에 확장 적용
- Q: 추가 UI 라이브러리(shadcn/ui 등) 사용 여부? → A: 순수 Tailwind CSS만 사용
- Q: 대시보드 레이아웃 패턴 — 단일 카드 vs 멀티 섹션? → A: 단일 페이지에 폼 카드 + 진행상태 섹션 + 오디오 플레이어 섹션을 세로로 배치 (단일 컬럼)
