# Feature Specification: 듀얼 미디어 뷰 (원본 + 더빙 비교 재생)

**Feature Branch**: `feat/#17-dual-media-view`
**Created**: 2026-03-19
**Status**: Draft
**Input**: User description: "원본 동영상, 더빙 동영상 두개 화면 보기 기능 서비스에 추가. 원본 영상과 더빙 동영상을 같이 볼 수 있어야 함. 오디오일 경우도 원본 오디오와 더빙 오디오를 같이 볼 수 있게. 현재 영상도 오디오 파일로 나오는 문제 수정 포함."

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 영상 파일 업로드 시 비디오로 재생 (Priority: P1)

사용자가 영상 파일(mp4, mov, webm)을 업로드하고 더빙을 완료하면, 결과 화면에서 원본 영상이 비디오 플레이어로 재생된다. 현재는 영상 파일을 업로드해도 결과가 오디오 플레이어로만 표시되어 원본 영상을 볼 수 없는 문제를 해결한다.

**Why this priority**: 핵심 버그 수정. 영상을 올렸는데 오디오로만 나오는 것은 사용자 기대와 완전히 다른 경험이므로 최우선으로 해결해야 한다.

**Independent Test**: 영상 파일을 업로드하고 더빙 완료 후, 원본 영상이 비디오 플레이어에서 재생되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 mp4 영상 파일을 업로드하고 더빙을 완료함, **When** 결과 화면이 표시됨, **Then** 원본 영상이 비디오 플레이어로 재생된다 (오디오 플레이어가 아닌)
2. **Given** 사용자가 mov/webm 영상 파일을 업로드함, **When** 더빙 파이프라인이 완료됨, **Then** 해당 포맷의 원본 영상도 비디오 플레이어로 올바르게 재생된다

---

### User Story 2 - 파일 업로드 직후 원본 미디어 미리보기 (Priority: P1)

사용자가 파일을 업로드하면, 더빙 파이프라인이 실행되기 전에 원본 미디어를 미리보기로 확인할 수 있다. 영상 파일은 비디오 플레이어로, 오디오 파일은 오디오 플레이어로 즉시 표시된다.

**Why this priority**: 사용자가 올바른 파일을 업로드했는지 즉시 확인할 수 있어야 하며, 업로드 직후 미디어 타입에 맞는 플레이어를 제공하는 것이 자연스러운 경험이다.

**Independent Test**: 영상/오디오 파일을 업로드한 직후, 더빙 시작 전에 원본 미디어가 적절한 플레이어로 재생되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 mp4 영상 파일을 선택함, **When** 파일이 업로드됨, **Then** 더빙 시작 전에 원본 영상이 비디오 플레이어로 미리보기 재생된다
2. **Given** 사용자가 mp3 오디오 파일을 선택함, **When** 파일이 업로드됨, **Then** 더빙 시작 전에 원본 오디오가 오디오 플레이어로 미리보기 재생된다

---

### User Story 3 - 원본 영상과 더빙 오디오 나란히 비교 (Priority: P1)

더빙 완료 후, 사용자는 원본 영상(비디오 플레이어)과 더빙된 오디오(오디오 플레이어)를 나란히 볼 수 있다. 데스크톱에서는 좌우 배치, 모바일에서는 상하 배치로 표시된다.

**Why this priority**: 사용자가 요청한 핵심 기능. 원본과 더빙 결과를 함께 보는 것이 이 기능의 주요 가치이다.

**Independent Test**: 영상 업로드 → 더빙 완료 후 화면에 원본 비디오 플레이어와 더빙 오디오 플레이어가 동시에 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 영상 파일로 더빙이 완료됨, **When** 데스크톱에서 결과 화면을 봄, **Then** 원본 영상 플레이어와 더빙 오디오 플레이어가 좌우로 나란히 표시된다
2. **Given** 영상 파일로 더빙이 완료됨, **When** 모바일에서 결과 화면을 봄, **Then** 원본 영상 플레이어와 더빙 오디오 플레이어가 상하로 쌓여 표시된다
3. **Given** 결과 화면에서 원본 영상과 더빙 오디오가 표시됨, **When** 각각 독립적으로 재생/일시정지를 조작함, **Then** 각 플레이어가 개별적으로 제어된다

---

### User Story 4 - 원본 오디오와 더빙 오디오 나란히 비교 (Priority: P2)

사용자가 오디오 파일(mp3, wav 등)을 업로드하고 더빙을 완료하면, 원본 오디오 플레이어와 더빙 오디오 플레이어를 나란히 볼 수 있다.

**Why this priority**: 오디오 파일 사용자에게도 동일한 비교 경험을 제공하지만, 영상 버그 수정보다 우선순위가 낮다.

**Independent Test**: 오디오 파일 업로드 → 더빙 완료 후 원본 오디오와 더빙 오디오 플레이어가 함께 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 오디오 파일(mp3)로 더빙이 완료됨, **When** 결과 화면을 봄, **Then** 원본 오디오 플레이어와 더빙 오디오 플레이어가 나란히 표시된다
2. **Given** 두 오디오 플레이어가 표시됨, **When** 원본 재생 버튼을 누름, **Then** 원본 오디오만 재생되고 더빙 오디오는 영향받지 않는다

---

### User Story 5 - 원본과 더빙 미디어 레이블 구분 (Priority: P3)

결과 화면에서 어떤 플레이어가 원본이고 어떤 것이 더빙인지 명확하게 레이블로 구분된다.

**Why this priority**: 두 플레이어가 나란히 있을 때 혼동을 방지하는 기본적인 UX 요소이다.

**Independent Test**: 결과 화면에서 각 플레이어에 "원본" / "더빙" 레이블이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 더빙 완료 후 결과 화면이 표시됨, **When** 두 플레이어를 확인함, **Then** 각각 "원본", "더빙" 등 명확한 레이블이 표시된다

---

### Edge Cases

- 사용자가 영상 파일을 업로드했지만 브라우저가 해당 영상 포맷을 지원하지 않는 경우 오류 메시지를 표시한다
- 원본 파일이 매우 긴 경우(예: 10분 이상) 원본 Blob URL 메모리 관리에 주의한다
- 모바일 화면에서 두 플레이어를 상하로 쌓아 표시하여 공간 문제를 해결한다
- 반복 더빙(재제출) 시 이전 원본 Blob URL이 올바르게 해제되어야 한다
- 파일 업로드 후 미리보기 중 사용자가 파일을 변경하면, 이전 미리보기 Blob URL을 해제하고 새 파일로 교체한다

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 시스템은 업로드된 파일의 MIME 타입을 기반으로 영상(video/*)과 오디오(audio/*)를 구분해야 한다
- **FR-002**: 영상 파일이 업로드된 경우, 원본 파일을 비디오 플레이어로 재생해야 한다 (오디오 플레이어가 아닌)
- **FR-003**: 파일 업로드 직후, 더빙 시작 전에 원본 미디어를 미리보기로 표시해야 한다
- **FR-004**: 더빙 완료 후, 원본 미디어(영상 또는 오디오)와 더빙 오디오를 나란히 표시해야 한다
- **FR-005**: 오디오 파일이 업로드된 경우, 원본 오디오와 더빙 오디오를 나란히 표시해야 한다
- **FR-006**: 원본 미디어의 Blob URL을 생성하고 세션 내에서 유지해야 하며, 새 제출 또는 파일 변경 시 이전 URL을 해제해야 한다
- **FR-007**: 각 플레이어에 "원본" / "더빙" 레이블을 표시하여 구분해야 한다
- **FR-008**: 각 플레이어는 독립적으로 재생/일시정지/탐색이 가능해야 한다
- **FR-009**: 데스크톱에서는 두 플레이어를 좌우 나란히, 모바일에서는 상하로 쌓아 표시해야 한다
- **FR-010**: 다운로드 기능은 더빙 결과에만 제공한다 (원본 미디어 다운로드 불필요)

### Key Entities

- **원본 미디어 (Source Media)**: 사용자가 업로드한 원본 파일. 미디어 타입(video/audio), Blob URL, 파일명을 포함
- **더빙 결과 (Dubbed Result)**: TTS 파이프라인에서 생성된 오디오. Blob URL로 관리되며 항상 오디오 형식

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 영상 파일 업로드 시 100% 비디오 플레이어로 재생된다 (오디오 플레이어로 렌더링되는 경우 0건)
- **SC-002**: 더빙 완료 후 원본과 더빙 결과가 동시에 화면에 표시된다
- **SC-003**: 사용자가 3초 이내에 원본과 더빙 결과를 구분할 수 있다 (레이블 제공)
- **SC-004**: 반복 더빙 시 이전 Blob URL이 정상 해제되어 메모리 누수가 발생하지 않는다
- **SC-005**: 파일 업로드 직후 1초 이내에 원본 미디어 미리보기가 표시된다

### User Story 6 - 영상 더빙 시 영상 형태로 결과 제공 (Priority: P1)

사용자가 영상 파일을 업로드하고 더빙을 완료하면, 더빙된 오디오가 원본 영상에 합성되어 **영상 형태**로 결과가 제공된다. 현재는 TTS API가 오디오(MP3)만 반환하므로, 클라이언트에서 FFmpeg.wasm을 사용하여 원본 영상의 비디오 트랙 + 더빙 오디오를 합성한 새 영상 파일을 생성한다.

**Why this priority**: 영상을 더빙했는데 오디오 파일로만 결과가 나오는 것은 사용자 기대와 맞지 않는다. 더빙 결과도 영상 형태여야 자연스러운 경험이다.

**Independent Test**: 영상 파일 업로드 → 더빙 완료 → 더빙 결과가 비디오 플레이어(영상)로 재생되며, 다운로드 시 영상 파일로 저장되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 영상 파일(mp4)로 더빙이 완료됨, **When** 결과 화면을 봄, **Then** 더빙 결과가 비디오 플레이어에서 영상으로 재생된다 (오디오 플레이어가 아닌)
2. **Given** 영상 더빙이 합성 중임, **When** FFmpeg.wasm이 영상+오디오를 머징함, **Then** 사용자에게 "영상 합성 중..." 진행 상태가 표시된다
3. **Given** 영상 더빙 결과가 영상으로 제공됨, **When** 다운로드 버튼을 클릭함, **Then** 영상 파일(.mp4)로 다운로드된다
4. **Given** 오디오 파일(mp3)로 더빙이 완료됨, **When** 결과 화면을 봄, **Then** 기존과 동일하게 오디오 플레이어로 결과가 표시된다 (FFmpeg 합성 불필요)

---

### Edge Cases

- 사용자가 영상 파일을 업로드했지만 브라우저가 해당 영상 포맷을 지원하지 않는 경우 오류 메시지를 표시한다
- 원본 파일이 매우 긴 경우(예: 10분 이상) 원본 Blob URL 메모리 관리에 주의한다
- 모바일 화면에서 두 플레이어를 상하로 쌓아 표시하여 공간 문제를 해결한다
- 반복 더빙(재제출) 시 이전 원본 Blob URL이 올바르게 해제되어야 한다
- 파일 업로드 후 미리보기 중 사용자가 파일을 변경하면, 이전 미리보기 Blob URL을 해제하고 새 파일로 교체한다
- FFmpeg.wasm 로딩 실패 시 오디오 플레이어 폴백으로 더빙 결과를 제공한다
- 큰 영상 파일(수백 MB)은 브라우저 메모리 제한에 걸릴 수 있으므로 적절한 에러 처리가 필요하다
- FFmpeg.wasm 영상 합성 중 사용자가 새 파일을 업로드하면 진행 중인 합성을 취소하고 새 파일로 시작한다

- **FR-012**: 영상 합성 진행 상태("영상 합성 중...")를 사용자에게 표시해야 한다
- **FR-013**: 영상 더빙 결과는 비디오 플레이어로 재생되고, 영상 파일(.mp4)로 다운로드 가능해야 한다
- **FR-014**: FFmpeg.wasm 로딩 또는 합성 실패 시 오디오 플레이어 폴백으로 결과를 제공해야 한다

### Key Entities

- **원본 미디어 (Source Media)**: 사용자가 업로드한 원본 파일. 미디어 타입(video/audio), Blob URL, 파일명을 포함
- **더빙 결과 (Dubbed Result)**: TTS 파이프라인에서 생성된 오디오. 영상 입력 시 FFmpeg.wasm으로 영상 합성 후 비디오 Blob URL로 관리. 오디오 입력 시 기존대로 오디오 Blob URL로 관리

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 영상 파일 업로드 시 100% 비디오 플레이어로 재생된다 (오디오 플레이어로 렌더링되는 경우 0건)
- **SC-002**: 더빙 완료 후 원본과 더빙 결과가 동시에 화면에 표시된다
- **SC-003**: 사용자가 3초 이내에 원본과 더빙 결과를 구분할 수 있다 (레이블 제공)
- **SC-004**: 반복 더빙 시 이전 Blob URL이 정상 해제되어 메모리 누수가 발생하지 않는다
- **SC-005**: 파일 업로드 직후 1초 이내에 원본 미디어 미리보기가 표시된다
- **SC-006**: 영상 파일 더빙 시 더빙 결과가 영상 형태(비디오 플레이어)로 제공된다

## Assumptions

- TTS API는 항상 오디오(MP3) 형식을 반환한다 (ElevenLabs TTS API 출력)
- 영상 입력 시 클라이언트에서 FFmpeg.wasm을 사용하여 원본 영상 + 더빙 오디오를 합성한다 (서버 불필요)
- 원본 파일은 클라이언트 메모리(Blob URL)에서만 관리되며, 서버에 별도 저장하지 않는다
- 브라우저의 기본 비디오/오디오 플레이어 컨트롤을 활용한다
- 두 플레이어 간 동기화 재생(sync playback)은 이번 범위에 포함하지 않는다
- 다운로드 기능은 더빙 결과에만 제공하며, 원본 미디어 다운로드는 제공하지 않는다
- FFmpeg.wasm은 SharedArrayBuffer 또는 단일 스레드 모드로 동작한다
